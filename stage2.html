Stage chez LEMNIA

Ce stage, réalisé chez LEMNIA, avait pour objectif de me former
aux outils utilisés par l'entreprise et de mettre en place
des infrastructures réseau professionnelles.
Il s'est déroulé sur plusieurs semaines avec des missions variées :
téléphonie IP, proxy web, cluster et pare-feu OPNsense.

Semaine 1 — Formation 3CX

La première semaine avait pour objectif de me former aux outils utilisés par LEMNIA.
Lundi : formation via les webinars officiels 3CX (hébergement, sauvegardes, mises à jour, rôles utilisateurs).
Mardi / Mercredi : suite de la formation 3CX et prise en main du logiciel — création d'un réseau de téléphonie IP avec des téléphones Yealink.
Jeudi : formation à ARTICA Protection via vidéo YouTube et documentation officielle.
Vendredi : prise en main de Proxmox, le lab virtuel interne de l'entreprise.

Semaine 2 — Proxy ARTICA (1/2)

Création d'un proxy web lié à un Active Directory avec ARTICA.
Étape 1 : création des zones DNS dans le gestionnaire DNS Windows (zone directe stage.local et zone inverse 192.168.101.x).
Étape 2 : intégration du serveur proxy dans le DNS — ajout d'un enregistrement de type A pour artica-proxy (192.168.101.10) avec mise à jour du pointeur PTR associé.
Étape 3 : intégration du serveur proxy au domaine via le DNS primaire (192.168.101.15).

Semaine 3 — Proxy ARTICA (2/2)

Étape 4 : connexion au compte administrateur de l'AD — liaison Kerberos avec AD.stage.local (Windows Server 2019), compte tdefournier@stage.local.
Étape 5 : configuration du proxy sur le client Windows — adresse http://artica-proxy.stage.local port 3128.
Étape 6 : le proxy bloque correctement les sites non autorisés (Cache Access Denied via Artica Proxy 4.50).

Semaine 4 — Cluster HaCluster (1/2)

Mise en place d'un cluster ARTICA et réplication du proxy.
Étape 1 : création d'un serveur HaCluster depuis l'interface ARTICA — choix du mode HaCluster parmi les activités disponibles (Service Proxy, Pare-feu DNS, DNS Cache, Reverse-Proxy…).
Étape 2 : liaison avec le serveur Active Directory — authentification Kerberos, configuration du FQDN (dc1.domain.tld), suffix LDAP, synchronisation de l'heure avec l'AD.
Étape 3 : liaison avec les serveurs proxy maître (artica-proxy1 — 192.168.101.21) et esclave (artica-proxy2 — 192.168.101.22) via les ports 8090, 8091, 58787 et 9000.

Semaine 4 — Cluster HaCluster (2/2)

Étape 4 : le cluster est opérationnel — le proxy bloque les sites correctement avec le message "Cache Access Denied" généré par artica-proxy1.stage.local.

Difficultés rencontrées :
— Impossibilité de connecter le serveur HaCluster à l'Active Directory.
— Réplication du proxy qui ne fonctionnait pas.
— Règle de filtrage qui ne fonctionnait pas.

Solutions apportées :
— Mise à jour du noyau du serveur vers une version plus récente.
— Recréation d'un proxy maître et esclave.
— Réinitialisation du client test.

Semaine 5 — OPNsense (1/2)

Contexte : réaliser une infrastructure réseau avec pare-feu OPNsense, deux VLANs (VLAN 200 → 192.168.200.0/24 et VLAN 201 → 192.168.201.0/24), un serveur web, un PC filtré par ARTICA et un AD.
Étape 1 : installation d'OPNsense 24.1 — mode UFS GPT/UEFI Hybrid.
Étape 2 : configuration initiale d'OPNsense via le menu console (Boot Multi user).
Étape 3 : assignation des interfaces réseau — vtnet0 (WAN), vtnet1 (OPT1/VLAN 200), vtnet2 (OPT2/VLAN 201).

Semaine 5 — OPNsense (2/2)

Étape 4 : assignation des adresses IP — OPT1 (vtnet1_vlan200) → 192.168.200.1/24, OPT2 (vtnet2_vlan201) → 192.168.201.1/24, WAN → 192.168.101.16/24.
Étape 5 : activation des interfaces dans l'interface web OPNsense — configuration de l'adresse IPv4 statique 192.168.200.1/24 sur OPT1, passerelle auto-détectée.

Objectifs de l'infrastructure :
— Le client (CL1) accède au port 80 du serveur web.
— Le PC doit être filtré par ARTICA en mode Connecté.
— Le serveur web ne peut pas atteindre le VLAN 201.
— Le PC peut accéder au serveur web.
